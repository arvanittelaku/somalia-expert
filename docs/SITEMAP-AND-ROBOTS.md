# Sitemap and robots.txt

This document describes how sitemaps and `robots.txt` are produced, which source is canonical in production, and how the project keeps them in sync with the app.

## Overview

| Output | Path served at | Production source | Committed file (CI) |
|--------|----------------|-------------------|---------------------|
| XML sitemap | `/sitemap.xml` | **`app/sitemap.ts`** (Next.js route) | `public/sitemap.xml` |
| robots.txt | `/robots.txt` | `public/robots.txt` (static) | `public/robots.txt` |

Both sitemap outputs are built from the same URL inventory (`buildPublicUrlInventory` in `lib/seo/publicUrlInventory.ts`). Crawlers should use a **single** sitemap URL only.

## Canonical sitemap in production

**`app/sitemap.ts` is the canonical sitemap** served at:

`https://www.africaexpertwitness.com/sitemap.xml`

Next.js generates this route at build/runtime from `buildPublicUrlInventory()`. When both `app/sitemap.ts` and `public/sitemap.xml` exist, the App Router route takes precedence in production — crawlers receive the dynamic sitemap, not a stale static copy.

**`public/sitemap.xml`** is still generated by `npm run seo:generate` (also run in `prebuild`) so that:

- CI can verify the URL list without a running server (`npm run seo:verify`)
- The committed file documents the expected indexable URLs in the repo
- `lastmod` dates are refreshed before each build

**Rule:** After changing routes or slugs, run `npm run seo:generate` and commit `public/sitemap.xml` if URLs changed. `seo:verify` fails if `public/sitemap.xml` does not match the inventory — the same inventory `app/sitemap.ts` uses, so production and CI stay aligned.

## robots.txt — single Sitemap line

`public/robots.txt` is generated by `scripts/generate-seo.ts` and must declare **one** sitemap URL:

```
Sitemap: https://www.africaexpertwitness.com/sitemap.xml
```

That URL is the Next.js sitemap route above. Do not add alternate sitemap paths (e.g. `sitemap-0.xml` or a duplicate static file URL).

There is no `app/robots.ts`; static `public/robots.txt` is served as-is.

## URL inventory (`buildPublicUrlInventory`)

The script and `app/sitemap.ts` import `buildPublicUrlInventory` from `lib/seo/publicUrlInventory.ts`. That function assembles all indexable paths, then maps them to absolute URLs with the canonical host **https://www.africaexpertwitness.com**.

The inventory merges:

1. **Static app routes** — `APP_STATIC_PATHS` (home, services, regions, countries hub, expertise hub, case types, guides, FAQ, glossary, experts, qualifications, fees, how-to-instruct, what-is).
2. **Dynamic routes** — Slugs from `data/regions.ts`, `data/countries.ts`, `data/expertise-areas.ts`, `data/case-types.ts`, and `data/guides.ts`.

**Excluded from the sitemap** (still valid pages, not for indexation lists):

- `/contact`
- `/thank-you`
- `/privacy`
- `/terms`

The combined list is deduplicated, sorted, and exposed as `allUrls` (absolute) and `allPaths` (pathname only).

## Sitemap generation (`scripts/generate-seo.ts`)

For each URL in the inventory, the script emits a standard sitemaps.org `<url>` entry with:

- `<loc>` — Absolute URL
- `<lastmod>` — Calendar date when generation runs (UTC `YYYY-MM-DD`)
- `<changefreq>` — From inventory (home `weekly`, most pages `monthly`)
- `<priority>` — From inventory (e.g. home `1.0`, country pages `0.88`)

The document is written to `public/sitemap.xml`. The `public/` directory is created if needed.

## robots.txt generation (`scripts/generate-seo.ts`)

`robots.txt` is a fixed template in the same script (`renderRobots()`), not derived from the URL inventory. It:

- Allows all crawlers on `/` by default
- **Disallows:** `/thank-you`, `/api/`
- Declares: `Sitemap: https://www.africaexpertwitness.com/sitemap.xml` (once)

The file is written to `public/robots.txt`.

## Commands

From the repo root:

```bash
npm run seo:generate   # Regenerate public/sitemap.xml + public/robots.txt
npm run seo:verify     # Fail if public/sitemap.xml <loc> entries differ from inventory
```

`seo:generate` runs automatically before `npm run build` via the `prebuild` script.

Run `seo:generate` after:

- Adding a static route to `APP_STATIC_PATHS`
- Adding slugs to any `data/*.ts` file used by dynamic routes
- Changing robots disallow rules in `publicUrlInventory.ts`
- Before release when you want fresh `lastmod` dates in `public/sitemap.xml`

`seo:verify` ensures the **committed** `public/sitemap.xml` matches `buildPublicUrlInventory()`. It does not fetch the live `/sitemap.xml` route, but both must list the same URLs because they share the same inventory function.

## Continuous integration

`.github/workflows/seo-checks.yml` runs on pull requests and manual dispatch:

1. `npm ci`
2. `npm run seo:generate`
3. `npm run seo:verify`
4. Fails if committed `public/sitemap.xml` or `public/robots.txt` differ from freshly generated output

## Operational notes

- **Canonical host** is `SITE_URL` in `lib/constants.ts` and the robots `Sitemap:` line. If the domain changes, update `lib/constants.ts` and regenerate.
- **New static pages** must be added to `APP_STATIC_PATHS` or `seo:verify` will fail after the next generate.
- **New dynamic pages** require a slug in the relevant `data/*.ts` file; the inventory reads those arrays automatically.
- **Do not** maintain a second hand-edited sitemap; keep `app/sitemap.ts`, `public/sitemap.xml`, and the inventory in sync via `seo:generate` + `seo:verify`.
